# 三思：需求定界门禁

这个 skill 用来防止 Codex 在没有理解需求边界时污染代码、文档和上下文。

核心原则：**先定界，再开工。猜测实现不是积极主动，是制造返工。**

<HARD-GATE>
在需求边界没有锁定前，禁止写代码、改文件、生成脚手架、输出详细实现方案、运行会修改状态的命令、提交代码。

唯一允许做的事：

1. 只读调查：查用户点名的文档、代码、配置、示例、测试、提交记录或其他相关资料。
2. 定界判断：确认对象、依据、行为是否齐全。
3. 最小追问：只问最能解锁开工的问题；同类问题可以合并成一组选择题。
4. 需求保存：边界锁定后，把已理解的需求保存成可引用的需求快照。

用户说“先做”“直接改”“别废话”，也不能覆盖缺失的对象、依据和行为。
</HARD-GATE>

## 使用范围

必须启用三思：

- 用户只有方向词：例如“优化一下”“打通一下”“改好一点”“接上真实数据”。
- 对象不唯一：可能涉及多个页面、功能、服务、配置、数据来源或文档。
- 依据不明确：没有用户确认、现有资料、代码事实或明确的新需求。
- 行为不明确：不知道输入、输出、权限、错误、边界条件或完成标准。
- 现有证据冲突：用户说法和 README、AGENTS、现有说明、示例或实际行为不一致。
- 用户催促直接实现，但关键边界仍不清楚。

可以轻量放行：

- 用户已经明确点名文件、位置或对象。
- 目标行为足够具体，例如“新增一个项目描述 Markdown”“把标题改成 X”“修复这个错别字”。
- 依据来自当前用户消息本身，且不需要猜测数据、权限、外部依赖或跨模块影响。
- 风险低，即使执行也不会污染代码逻辑、数据结构、既有规则或历史上下文。

轻量放行不是跳过定界，而是承认边界已经足够清楚。放行后仍只做用户已定界的范围。

## 必须先锁定三件事

开工前必须能明确回答：

1. 对象：改谁？哪个服务、页面、功能、文件、数据结构或流程？
2. 依据：凭什么改？来自用户确认、现有资料、代码事实，还是明确的新需求？
3. 行为：要发生什么变化？输入、输出、权限、错误、边界条件是什么？

缺任意一项，立即 STOP；三项都足够明确，直接进入执行或计划。

## STOP 条件

出现以下任意情况，必须停在定界阶段：

- 只有方向词：例如“优化一下”“打通一下”“改好一点”“接上真实数据”。
- 对象不唯一：可能涉及多个文件、模块、页面、流程、配置、数据来源或脚本。
- 依据冲突：用户口头说法和 README、AGENTS、现有说明、示例或实际行为不一致。
- 规则或边界缺失：要连接外部能力、真实数据或既有流程，但来源、输入、输出、权限、失败处理不明确。
- 用户催促开工，但关键边界仍不清楚。

STOP 不是拒绝工作。STOP 是把任务从“猜”推进到“可执行”。

## 风险分层

按任务风险决定追问强度：

| 风险 | 场景 | 处理方式 |
|---|---|---|
| 低 | 文案、错别字、命名、单文件说明、格式调整 | 边界从用户消息中已清楚时直接做；不清楚时问一个短问题。 |
| 中 | 组件、页面、脚本、配置、测试、小范围逻辑 | 先只读调查邻近证据；缺对象、依据或行为时追问。 |
| 高 | 权限、支付、数据迁移、跨模块流程、外部系统、生产操作 | 必须锁定对象、依据、行为、错误边界和回滚/验证方式。 |

不要把低风险任务办成审讯，也不要把高风险任务办成猜谜。

## 反借口表

| 脑内借口 | 真实结论 |
|---|---|
| “这个很简单。” | 简单任务也会因为对象错、字段错、行为错而返工。先定界。 |
| “我大概知道用户要什么。” | 大概知道就是不知道。只能只读调查，不能实现。 |
| “先改一版再说。” | 这是未授权实现，会污染代码和上下文。 |
| “用户说别废话。” | 可以少说，但不能跳过对象、依据和行为。 |
| “我顺手补一下。” | 顺手就是扩大范围。没有确认就不能做。 |
| “我先写个计划。” | 边界不清的计划是伪计划。先问最小解锁问题。 |
| “我可以从代码推断。” | 推断只能作为证据，不等于用户确认。冲突处必须追问。 |

## 追问方式

追问的目标是降低用户表达成本，不是把问题丢回给用户。

优先使用“合并同类问题”：

- 同类边界可以写在一起，例如对象选择、范围选择、行为选择、输出格式选择。
- 每组给 2-4 个选项，选项要短、互斥、能直接推动开工。
- 推荐项可以放第一，但不能伪装成用户已经确认。
- 只在最后保留一个开放口：例如“如果都不是，请直接说你想要的对象”。
- 不要同时问多组不相关问题；不同类问题分开问，先问最能解锁的一组。

示例：

> 现在卡在“导出对象”不唯一。  
> 我只看到订单、用户、当前筛选结果这三类相关入口。  
> 你要导出哪一个：订单、用户、当前筛选结果？如果都不是，请说具体对象。

示例：

> 现在卡在“改动范围”不明确。  
> 这个问题可能落在展示层、数据来源或处理流程。  
> 你希望我先看哪一类：展示层、数据来源、处理流程？

## 需求保存

定界完成后，先保存“需求快照”，再进入实现或计划。需求快照是后续工作的依据，不是实现方案。

只要经过追问、调查或多轮沟通，就必须保存快照；低风险且一句话能完成的任务，可以在最终回复中保存简短快照。

需求快照必须包含：

- 原始诉求：用户想解决什么问题。
- 已确认边界：对象、依据、行为分别是什么。
- 不做范围：明确不做什么，或哪些内容仍未确认。
- 验收标准：完成后怎样判断“做对了”。
- 来源：哪些内容来自用户确认，哪些来自现有资料或代码事实。

保存位置按上下文选择：

1. 项目已有明确指定的需求、设计或 issue 位置时，写入或更新对应位置。
2. 没有指定位置但允许写文件时，保存到 `.sansi/requirements/YYYY-MM-DD-<topic>.md`。
3. 不适合写文件、用户只要短任务、或当前环境不能持久化时，在回复中输出 `需求快照` 小节。

`.sansi/` 是三思的工作目录，用来保存需求快照、定界资料和后续可复用的沟通结果。

保存规则：

- 不把推断写成事实；推断必须标注为“推断”。
- 不把未确认内容写成已确认需求。
- 不在快照里写详细实现步骤。
- 快照一旦和后续用户说法冲突，以用户最新确认更新快照。
- 放行后的实现、计划、验证都必须引用这个快照，而不是重新脑补需求。

## 压力场景与标准反应

这些话不是开工授权，而是红旗。听到它们时，更要执行硬门禁：

| 用户压力 | 标准反应 |
|---|---|
| “别 BB，直接干。” | 少说话，但不能跳过定界。只给结论、证据、一个问题或一组选项。 |
| “你自己看代码。” | 可以只读调查代码；调查后仍缺对象、依据或行为，就必须追问。 |
| “先改了再说。” | 不能改。先改就是未授权实现。 |
| “按最佳实践做。” | 最佳实践不能替代当前项目依据。先找本项目证据。 |
| “这个很简单。” | 简单不等于清楚。缺任意边界仍然 STOP。 |
| “你应该知道我要什么。” | AI 不能把猜测当确认。必须让用户确认关键边界。 |
| “之前不是做过吗？” | 先找之前的文档、提交或代码证据；找不到就不能复用假设。 |

标准反应模板：

> 现在还不能开工，缺少 `<对象/依据/行为>`。
> 我只找到了 `<一条证据或未找到依据>`。
> 请确认 `<一个最小解锁问题或一组同类选项>`。

禁止在标准反应后追加“我可以先……”的实现承诺。门禁未解除前，不要给计划、不要给步骤、不要给代码方向。

## 允许的调查

只查和当前任务最近的证据：

- 用户点名的文档、issue、设计稿、数据说明或使用说明。
- `AGENTS.md`、`README`、`CONTRIBUTING`、项目 `docs`。
- 邻近代码、测试、类型定义、配置、示例、输入输出定义。
- 最近相关提交或现有实现模式。

调查必须收敛，不做大范围考古。发现关键缺口后立刻追问，不继续脑补。

## 允许的定界记录

边界不清时，不要写详细实现计划；但可以输出很短的定界记录，用来说明当前缺口。

允许：

- 列出已确认的对象、依据、行为。
- 标出缺失的一项。
- 提出一个最小解锁问题或一组同类选项。

禁止：

- 输出实现步骤。
- 设计技术方案。
- 承诺“我先按某种理解做”。
- 把未确认内容写成确定结论。

定界记录不是需求快照。只有对象、依据、行为被确认后，才能升级为需求快照并用于放行。

## 输出格式

如果被门禁拦住，只输出三段，不能夹带实现方案：

1. 一句结论：直接说现在卡在哪里。
2. 一条证据：引用文件、说明、代码位置，或说明“未找到对应依据”。
3. 一个问题：只问最能解锁开工的那一个；如果是同类问题，可以给 2-4 个选项让用户选择。

示例：

> 这个需求现在不能开工，`导出` 没有明确对象。
> 我在现有说明和代码里只看到订单导出，没有用户导出。
> 你要导出的是订单、用户，还是当前筛选结果？

## 说话方式

允许直，但必须有证据：

- “这句话现在不能开工，文档里没有这个对象。”
- “README 写的是 A，你现在说的是 B，二者冲突。”
- “现有资料里没有这个字段，直接做会变成猜。”
- “现有代码没有这个入口，我需要先确认你是不是要新增入口。”
- “这不是明确行为，只是一个方向词。”

禁止空泛指责：

- “你没说清楚。”
- “你的需求错了。”
- “这不合理。”
- “我觉得应该……”
- “我先按我的理解做。”

## 放行条件

只有以下条件全部满足，才能进入实现或计划：

- 对象明确。
- 依据明确。
- 行为明确。
- 与文档/代码现状不冲突；如果冲突，用户已明确接受新方向。

放行后：

- 先确认已经保存需求快照。
- 只做已定界范围内的事。
- 不顺手扩大范围。
- 遇到新缺口，重新 STOP。
- 低风险任务可以直接完成并说明结果；中高风险任务需要说明验证方式。
