# 三思：让 AI 在开工前先听懂

AI 不是不会干，很多时候，是还没有真正懂你的需求。

它擅长执行，也擅长顺从。人说一句“先做吧”，它往往就会立刻动手；人说一句“差不多这个意思”，它也可能把猜测当成确认。问题不在于 AI 不努力，而在于它太容易把未被说清的意图，误认成已经明确的指令。

而没有沟通清楚就开始干，代价往往很高。

表面上省下的是几分钟提问，后面付出的却可能是漫长的返工、反复的等待、被污染的上下文、跑偏的方案，以及一次又一次“这不是我要的”。在软件、文档、产品和自动化任务里，最贵的常常不是写代码，而是从错误理解里再绕回来。

“三思”要做的事情很简单：在开工前，多问一句，多核对一步。

它不是拖慢进度，而是把进度从猜测里救出来。它要求 AI 在真正动手之前，先确认三件事：

1. 对象：到底要改谁、做谁、处理谁？
2. 依据：凭什么这样做，来自用户确认、现有文档、代码事实，还是明确的新需求？
3. 行为：最终要发生什么变化，输入、输出、边界和异常分别是什么？

当这三件事没有说清楚时，“三思”会让 AI 停下来，只做调查，只问最能解锁问题的那一个问题。它相信，好的协作不是一味响应，而是在关键处敢于确认；不是把顺从当效率，而是把理解当开始。

因为真正可靠的 AI 协作，不是“你说什么我就做什么”，而是“我先确认我听懂了，再把事情做好”。
